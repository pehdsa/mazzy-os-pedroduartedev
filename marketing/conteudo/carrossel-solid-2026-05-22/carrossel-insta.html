Concepts 02 / 08
S
Single Responsibility
Responsabilidade Única
Uma classe, função ou módulo deve ter uma única responsabilidade.
Exemplo
Em vez de uma classe Relatorio que gera dados, formata PDF e envia email, separa em três:
GeradorRelatorio FormatadorPDF EnviadorEmail

Concepts 03 / 08
O
Open/Closed
Aberto pra Extensão, Fechado pra Modificação
Você deve adicionar comportamentos novos sem alterar o código que já funciona.
Exemplo
Em vez de editar uma função gigante com vários if pra cada tipo de desconto, cria estratégias:
DescontoClienteComum DescontoClienteVIP DescontoCupom

Concepts 04 / 08
L
Liskov Substitution
Substituição de Liskov
Uma classe filha deve poder substituir a classe pai sem quebrar o comportamento esperado.
Exemplo
Se Pato tem o método voar(), criar Pinguim extends Pato vai dar problema. Pinguim não voa.
O erro está no modelo, não no pinguim.

Concepts 05 / 08
I
Interface Segregation
Segregação de Interfaces
Não force uma classe a implementar métodos que ela não usa.
Exemplo
Em vez de uma interface Funcionario com programar(), vender() e limpar(), cria interfaces menores e específicas:
Programador Vendedor AuxiliarLimpeza

Concepts 06 / 08
D
Dependency Inversion
Inversão de Dependência
Dependa de abstrações, não de implementações concretas.
Exemplo
Em vez de uma classe depender de MySQLDatabase direto, ela depende da interface Database:
MySQL PostgreSQL MongoDB
Trocar de banco vira detalhe.

Concepts 07 / 08
Resumo prático
Pra que serve SOLID?
Não é sobre código "bonito". É sobre reduzir dor quando o sistema muda.
✓
Separar responsabilidades
✓
Evitar alterações perigosas
✓
Facilitar testes
✓
Reduzir acoplamento
✓
Tornar o código mais flexível
Se seu código é fácil de mudar, provavelmente está no caminho certo.
